# Архитектура Проекта (VPN Service)

Документация для разработчиков. В этом файле описано внутреннее устройство проекта, взаимодействие компонентов и основные принципы работы.

## 🏗 Обзор Архитектуры

Проект построен по принципу **Master-Node** и разделен на две логические части, которые могут работать как на одном сервере, так и на разных:

1. **Admin Server (Мастер-сервер)**
   - Выполняет роль единого центра управления (Control Plane).
   - Содержит базу данных (PostgreSQL), API-сервер (FastAPI) и Telegram-бота (Aiogram).
   - Управляет пользователями, их балансом, подписками и настройками маршрутизации.
2. **Node Server (VPN-Нода)**
   - Выполняет роль Data Plane (проксирование трафика).
   - Содержит Xray-core для обработки соединений (VLESS + Reality).
   - Содержит Python-агента (`agent/main.py`), который общается с Xray и Admin Server.

## 🔄 Взаимодействие Компонентов

### 1. Синхронизация Пользователей (Sync)
Каждые 10 минут агент ноды (`node_server/agent/main.py`) отправляет GET-запрос к API мастера: `/api/nodes/sync`.
Мастер возвращает список `uuid` активных пользователей (у которых баланс > -50 руб).
Агент перезаписывает файл `/app/xray/config.json`, вставляя туда полученные UUID, и перезапускает процесс Xray.

### 2. Подсчет Трафика и Биллинг (Stats)
Раз в 10 минут агент ноды опрашивает Xray API (`/usr/local/bin/xray api statsquery`) для получения статистики по потребленному трафику (downlink/uplink) для каждого UUID.
Собранная статистика отправляется POST-запросом на API мастера: `/api/nodes/stats`.
Мастер пересчитывает мегабайты в рубли (1 ГБ = 1 Рубль) и списывает средства с баланса пользователей в базе данных.
Если баланс уходит в минус, пользователю отправляется уведомление через Telegram-бота.

## 🗄 Структура Базы Данных

База данных работает на PostgreSQL + SQLAlchemy ORM.

Основные таблицы:
- `users`: Хранит Telegram ID, UUID (для Xray), баланс, статус (`is_active`), флаг маршрутизации (`split_tunneling`).
- `nodes`: Хранит IP и порты подключенных VPN-нод.
- `transactions`: История пополнений и списаний баланса.
- `promocodes`: Созданные промокоды.

*Миграции БД управляются с помощью Alembic (папка `admin_server/alembic`).*

## 🌐 API Эндпоинты (FastAPI)

Внутренние эндпоинты (защищены заголовком `X-Admin-API-Key`):
- `GET /api/nodes/sync` — отдает список активных UUID.
- `POST /api/nodes/stats` — принимает статистику трафика и запускает биллинг.

Публичные эндпоинты:
- `GET /sub/{uuid}` — генерирует конфигурацию подписки для клиента. В зависимости от параметра User-Agent отдает либо VLESS-ссылки, либо Sing-box JSON. Здесь же применяется логика Сплит-туннелинга (GeoSite/GeoIP ru), если у пользователя включен флаг `split_tunneling`.

## 💻 Локальная Разработка

Для развертывания проекта локально рекомендуется использовать Docker Compose.

1. Создайте `.env` файл в корне проекта, скопировав переменные из `admin_server/.env` (если они там есть) или задав базовые значения (BOT_TOKEN, ADMIN_API_KEY).
2. Вы можете использовать структуру `docker-compose.yml`, которую генерирует `install.sh`.
3. Запуск только мастера:
```bash
cd admin_server
docker build -t vpn_admin .
docker run -d --name vpn_admin -p 8000:8000 --env-file .env vpn_admin
```
*(Для полноценной локальной работы потребуется поднять PostgreSQL и Redis, лучше использовать docker-compose).*

## 🔐 Безопасность
- API между нодами и мастером защищено статичным `ADMIN_API_KEY`.
- Xray использует протокол VLESS с технологией REALITY, маскируя трафик под обычный HTTPS (TLS) запрос к белому сайту (задается в `REALITY_SHORT_ID` и `REALITY_PRIVATE_KEY`).
